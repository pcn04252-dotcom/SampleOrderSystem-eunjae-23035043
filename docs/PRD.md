## 1. 배경 및 문제 정의

### 1.1 배경 스토리

가상의 반도체 회사 **"S-Semi"** 는 다양한 종류의 반도체 시료(Sample)를 생산하여 연구소, 팹리스(Fabless) 업체, 대학 연구실 등의 고객에게 납품한다.

- 시료는 주문이 들어오면 웨이퍼 공정 설비를 통해 제작되고, 검수를 거쳐 고객에게 출고된다.
- 최근 주문량이 급증하면서 다음과 같은 문제가 발생했다.
  - "이 주문 처리됐나요?" — 주문 처리 상태를 추적할 수 없음
  - "공정 예약을 했는데, 언제 완성되는지 모르겠어요." — 생산 완료 시점 예측 불가
  - "이미 충분한 시료 재고가 있는데, 왜 추가 공정이 돌아가고 있나요?" — 재고와 생산 현황의 불일치
- 기존에는 엑셀과 메모장으로 주문을 관리하여 실수가 잦고, 재고·공정 현황을 한눈에 파악하기 어려웠다.

### 1.2 목표

체계적인 시료 관리를 위해 **주문 접수 → 승인/거절 → (필요 시) 생산 → 출고**로 이어지는 전체 흐름을 하나의 콘솔 시스템으로 관리하여, 주문 상태 추적, 생산 완료 예측, 재고-생산 정합성 문제를 해결한다.

### 1.3 이해관계자 및 역할 흐름

| 역할 | 설명 | 주요 행위 |
|---|---|---|
| 고객 (시료 요청자) | 필요한 시료를 이메일 등으로 요청하는 외부 주체 (시스템 사용자는 아님) | 시료 요청 |
| 주문 담당자 | 고객 요청에 맞게 주문서(주문 데이터)를 작성/관리 | 시료 등록, 주문 접수(예약) |
| 생산 담당자 (생산 라인 담당자) | 개발 시료 등록, 주문 수신 후 승인 또는 거절, 생산 관리 | 주문 승인/거절, 생산 현황 확인, 출고 처리 |

> 흐름: 고객 → (시료 요청) → 주문 담당자 → (주문서 전달) → 생산 담당자 → (승인/거절) → 주문 담당자에게 결과 회신

실제 콘솔 애플리케이션에서는 "주문 담당자"와 "생산 담당자"의 역할을 구분하는 로그인/권한 체계 없이, 하나의 콘솔에서 담당자가 메뉴를 선택해 각 역할의 작업을 순차적으로 수행하는 것으로 간주한다 (별도 인증 요구사항 없음).

## 2. 시스템 개요

### 2.1 아키텍처 원칙

- **인터페이스**: 콘솔(터미널) 기반. 담당자가 메뉴 번호 및 값을 직접 키보드로 입력하는 대화형(interactive) 방식.
- **핵심 도메인**: 시료(Sample), 주문(Order), 생산 라인(Production Line) 세 가지 엔티티를 중심으로 동작.
- **생산 라인**: 공장에서 시료 하나를 생산하는 설비의 흐름을 의미하는 개념적 단위.
  - 하나의 생산 라인은 한 번에 시료를 **하나씩** 생산한다 (단일 라인, 동시 병렬 생산 없음 — 예시 UI에서도 "생산라인 1개(단일 라인)"로 명시).
  - 생산 라인은 **주문이 들어온 시료에 대해서만** 생산한다 (사전 재고 축적을 위한 임의 생산 없음).
- 데이터는 애플리케이션 프로세스 내 메모리에 유지되는 것을 기본으로 하되, PoC 단계에서 정의한 데이터 영속성 방식(파일/JSON/DB 등)을 프로젝트 개발 단계에 통합하는 것을 권장한다 (본 PRD의 범위는 기능 명세이며, 영속성 기술 선택은 구현팀 재량).

### 2.2 전체 프로세스 플로우 (다이어그램 근거: 원본 PDF p.7)

```
[업무 단위: 접수]
  고객/주문담당자 → 주문 등록 (상태: RESERVED)
        │
        ▼
[업무 단위: 승인]
  생산담당자 → 승인 여부 판단
        │
   ┌────┴─────┐
   │거절       │승인
   ▼           ▼
REJECTED    재고 확인
            │
      ┌─────┴─────┐
      │재고 충분   │재고 부족
      ▼           ▼
  출고 준비      생산 요청
  (CONFIRMED)   (PRODUCING)
      │           │
      │       [생산 완료 시]
      │           │  (PRODUCING → CONFIRMED)
      │  ◄────────┘
      ▼
  출고 처리
  (RELEASED/RELEASE)
      │
      ▼
[업무 단위: 최종]
  고객 도달
```

> 원본 슬라이드 표기 불일치 주의: 플로우 다이어그램에서는 출고 완료 상태를 "RELEASED"로 표기했으나, 상태표(2.3절)에는 "RELEASE"로 표기되어 있다. 본 PRD에서는 **상태표 기준인 `RELEASE`를 공식 상태값**으로 채택한다. 구현 시 상태 enum/상수 이름은 `RELEASE`로 통일한다.

### 2.3 주문 상태(Order Status) 정의

모든 주문은 아래 5가지 상태 중 하나를 가진다.

| 상태 | 의미 | 진입 조건 |
|---|---|---|
| `RESERVED` | 주문 접수(예약) 완료, 승인 대기 중 | 고객 주문 생성 시 최초 상태 |
| `REJECTED` | 주문 거절됨 (정상 흐름 이탈 상태) | 생산 담당자가 RESERVED 주문을 거절 |
| `PRODUCING` | 주문 승인 완료 + 재고 부족으로 생산 진행 중 | 승인 시 재고 부족 판정 |
| `CONFIRMED` | 주문 승인 완료 + 출고 대기 중 (재고 확보됨) | 승인 시 재고 충분 판정, 또는 PRODUCING 상태에서 생산 완료 |
| `RELEASE` | 출고 완료 | 출고 처리 실행 |

**상태 전이 규칙:**

```
RESERVED --(거절)--------------------> REJECTED
RESERVED --(승인, 재고충분)-----------> CONFIRMED
RESERVED --(승인, 재고부족)-----------> PRODUCING
PRODUCING --(생산 완료)---------------> CONFIRMED
CONFIRMED --(출고 처리)---------------> RELEASE
```

- `REJECTED`는 정상 업무 흐름에서 제외되는 종결 상태이며, **모니터링 대상에서 제외**한다 (2.3절, 모니터링 기능 요구사항 참고).
- `RELEASE`는 최종 종결 상태이며 이후 상태 전이가 없다.
- 위 5개 상태 외의 임의 상태를 추가로 두지 않는다.

## 3. 기능 명세

시스템은 메인 메뉴에서 아래 5가지 기능 영역으로 진입한다.

| 메뉴 | 설명 |
|---|---|
| 1. 시료 관리 | 새로운 시료 등록, 목록 조회, 이름(속성) 검색 |
| 2. 주문 (접수/승인/거절) | 고객 주문 접수 및 생산 라인 담당자의 승인·거절 처리 |
| 3. 모니터링 | 상태별 주문 수 및 시료별 재고 현황 확인 |
| 4. 출고 처리 | CONFIRMED 상태 주문에 대한 출고 실행 |
| 5. 생산 라인 | 현재 생산 중인 시료 및 대기 중인 생산 큐 확인 |

### 3.1 메인 메뉴

**요구사항:**
- 기능별 선택 화면을 표시(Display)한다.
- 전체 시료에 대한 요약 정보(등록 시료 수, 총 재고, 전체 주문 건수, 생산 라인 대기 건수 등)를 상단에 함께 표시한다.
- 사용자가 번호를 입력하여 각 하위 메뉴로 진입한다.
- 화면 레이아웃/디자인은 자유롭게 구성 가능(예시 UI는 참고용).

**예시 요약 정보 항목** (자유 구성, 아래는 참고):
- 등록 시료 종수
- 총 재고 수량 합계
- 전체 주문 건수
- 생산 라인 대기 건수
- 현재 시각/시스템 상태

**입출력 예시(참고용):**
```
반도체 시료 생산주문관리 시스템
====================================================
시스템 현황  2026-04-16 09:32:15

등록 시료  12종     총 재고   2,840 ea
전체 주문  36건     생산라인  3건 대기
----------------------------------------------------
[1] 시료 관리          [2] 시료 주문
[3] 주문 승인/거절      [4] 모니터링
[5] 생산라인 조회       [6] 출고 처리
[0] 종료

선택 > _
```

### 3.2 시료 관리

**개념 정의:**
- 시료(Sample)는 시스템의 가장 기본이 되는 단위이다.
- 각 시료는 고유한 이름과 속성을 가지며, **시스템에 등록된 시료만 주문 가능**하다 (미등록 시료 ID로는 주문 생성 불가).

**시료 속성(필드) 정의:**

| 필드 | 설명 | 비고 |
|---|---|---|
| 시료 ID | 시료 고유 식별자 | 예: `S-001` 형식. 시스템 자동 채번 또는 입력값, 구현 시 결정 |
| 이름 | 시료명 | 예: "실리콘 웨이퍼-8인치" |
| 평균 생산시간 | 1개 생산에 걸리는 평균 시간(분 단위 등) | 생산 시간 계산에 사용 (3.6절 참고) |
| 수율 | (정상적인 시료 수) / (총 생산 시료 수), 0~1 사이 값 | 예: 100개 생산 중 정상 90개 = 0.9. 실 생산량 계산에 사용 |
| 현재 재고 | 현재 보유 중인 정상 시료 수량 | 조회 시 표시, 시료 자체 등록 속성은 아니고 파생/누적 값 |

**하위 기능:**

| 기능 | 설명 |
|---|---|
| 시료 등록 | 새 시료를 시스템에 추가. 입력값: 시료 ID, 이름, 평균 생산시간, 수율 (초기 재고는 0으로 시작하거나 별도 입력 가능하도록 구현 재량) |
| 시료 조회 | 등록된 모든 시료 목록을 표시. 각 시료의 현재 재고 수량을 함께 표시 |
| 시료 검색 | 이름 등 속성으로 특정 시료를 검색 (부분 일치 검색 등 구현 재량) |

**검증 규칙:**
- 시료 ID는 유일(unique)해야 하며, 중복 등록 시 오류 처리 또는 거부한다.
- 수율은 0 초과 1 이하 범위로 검증하는 것을 권장한다 (0이면 정상 생산 불가능한 상태이므로 실질적으로 무한 생산이 되어 로직 오류 유발 가능 — 구현 시 방어 로직 고려).
- 평균 생산시간은 0보다 큰 값이어야 한다.

**예시 UI(참고용):**
```
[1] 시료 관리
[1] 시료 등록  [2] 시료 목록  [3] 시료 검색  [0] 뒤로
선택 > 2

등록 시료 목록 (총 12종)
ID      시료명                평균 생산시간   수율    현재 재고
S-001   실리콘 웨이퍼-8인치    0.5 min/ea     0.92    480 ea
S-002   GaN 에피택셜-4인치     0.3 min/ea     0.78    220 ea
...
```

### 3.3 시료 주문 (접수)

**요구사항:**
- 고객이 시료를 요청하면 주문 담당자가 주문을 생성(예약)한다.
- 주문 생성 시 입력 값:
  - 시료 ID (등록된 시료여야 함)
  - 고객명
  - 주문 수량
- 주문 생성 직후, 해당 주문의 상태는 **`RESERVED`** 로 설정된다.
- 주문 생성 시 주문번호를 자동 채번한다 (예: `ORD-YYYYMMDD-NNNN` 형식).
- 이 시점에는 재고 확인/차감을 수행하지 않는다 — 재고 확인은 승인 단계(3.4절)에서 수행한다.

**검증 규칙:**
- 존재하지 않는 시료 ID로 주문 시도 시 오류 처리.
- 주문 수량은 1 이상의 정수여야 한다.

**예시 UI(참고용):**
```
[2] 시료 주문
시료 ID   > S-003
고객명    > 삼성전자 파운드리
주문 수량 > 200

입력 내용 확인
시료   SiC 파워기판-6인치 (S-003)
고객   삼성전자 파운드리
수량   200 ea

[Y] 예약 접수  [N] 취소
선택 > Y

예약 접수 완료.
주문번호  ORD-20260416-0043
현재 상태 RESERVED
```

### 3.4 주문 승인/거절

**요구사항:**

1. **접수된 주문 목록**
   - `RESERVED` 상태의 주문 목록을 표시한다.

2. **주문 승인**
   - 담당자가 목록에서 특정 주문을 선택하여 승인 처리한다.
   - 승인 시, **재고 상황에 따라 자동으로 2가지 방식 중 하나로 분기 처리**된다:
     - **재고 충분** (현재 재고 ≥ 주문 수량): 주문을 즉시 `CONFIRMED` 상태로 전환. 이때 해당 수량만큼 재고를 차감(예약 처리)한다.
     - **재고 부족** (현재 재고 < 주문 수량): 부족분에 대한 생산 작업을 생산 라인에 자동으로 등록(생산 큐에 추가)하고, 주문 상태를 `PRODUCING`으로 전환한다.
       - 부족분 = 주문 수량 − 현재 재고
       - 이때 현재 보유 중인 재고(있다면)는 우선 소진 처리하고, 부족분만큼만 생산을 요청한다.

3. **주문 거절**
   - 담당자가 목록에서 특정 주문을 선택하여 거절 처리한다.
   - 즉시 `REJECTED` 상태로 전환된다.
   - 거절된 주문은 재고에 영향을 주지 않는다.

**예시 UI(참고용):**
```

[1]  ORD-0041   LG이노텍        산화막 웨이퍼-SiO2   300 ea  RESERVED

### 3.5 모니터링

**요구사항:**

주문에 따른 시료 재고 정보를 모니터링하는 기능. 담당자가 시스템 현재 상태를 한눈에 파악할 수 있도록 구성한다.

1. **주문량 확인**
   - 상태별(`RESERVED` / `CONFIRMED` / `PRODUCING` / `RELEASE`) 주문 건수/목록을 확인한다.
   - `REJECTED`는 유효한 주문이 아니므로 모니터링 대상에서 **제외**한다.

2. **재고량 확인**
   - 각 시료별 현재 재고 수량을 확인한다.
   - 주문 대비 재고 수량에 따라 상태를 표기한다:

     | 표기 상태 | 조건 |
     |---|---|
     | 여유 | 주문 대비 재고 충분 |
     | 부족 | 주문 대비 재고 수량 부족 |
     | 고갈 | 재고 수량이 정확히 0인 상태 |

   - "주문 대비"의 기준(예: 미출고 주문 총 수요 대비 재고 비율)은 구현 시 구체적인 임계값/계산식을 정의해야 한다. 최소한 재고 0인 경우를 "고갈"로 명확히 구분해야 한다.

선택 > 1

### 3.6 생산 라인

**요구사항:**

1. **생산 로직**
   - 생산 라인은 주문량에 대한 부족분을 생산하되, 수율 및 오차를 고려하여 시료를 생산한다.
   - **실 생산량 계산식**: `실 생산량 = ceil(부족분 / 수율)`
     - 예: 부족분 170 ea, 수율 0.92 → ceil(170 / 0.92) = ceil(184.78...) = 185 (또는 예시 슬라이드 값 참고, 반올림 방식은 항상 올림(ceil) 적용)
   - **총 생산 시간 계산식**: `총 생산 시간 = 평균 생산시간 × 실 생산량`
   - 생산이 완료되면 해당 주문 상태를 `PRODUCING` → `CONFIRMED`로 자동 전환한다.
     - 생산 완료 시 생산된 정상 수량만큼 재고에 반영되며, 해당 주문에서 필요한 수량은 확보되어 출고 대기 상태가 된다.

2. **생산 현황 표기**
   - 현재 생산 중인 시료에 대한 정보를 표시한다.
   - 표기 정보 수준은 자율 결정 가능 (참고 예: 주문 정보, 현재까지의 생산 진행률/생산량, 완료 예정 시각 등).

3. **대기 주문 확인 (생산 큐)**
   - 생산 라인의 대기열(생산 큐)을 이용해 생산 작업을 대기 중인 목록을 출력한다.
   - **스케줄링 전략: FIFO** (First-In-First-Out) — 승인되어 생산이 필요해진 순서대로 처리한다.
   - 큐에 있는 각 대기 항목에 대해서도 부족분, 실생산량, 예상 완료 시각 등을 표시하는 것을 권장한다.

**핵심 계산 로직 요약:**

```
부족분      = max(0, 주문 수량 - 현재 재고)
실 생산량   = ceil(부족분 / 수율)
총 생산 시간 = 평균 생산시간 × 실 생산량
```

**예시 UI(참고용):**
```
[5] 생산라인 조회  FIFO 방식
생산라인 1개 (단일 라인)  현재 상태: RUNNING

현재 처리 중
  주문번호 ORD-20260416-0038  시료 SiC 파워기판-6인치
  주문량 80 ea  재고 30 ea → 부족 50 ea → 실생산량 54 ea (수율 0.92 / 49 min)
  진행 [========          ] 72%  완료 예정 09:49

대기 중인 주문 (FIFO 순)
순서  주문번호   시료                주문량   부족분   실생산량  예상 완료

## 6. 예외 처리 및 엣지 케이스 (구현 시 고려사항)

원본 PDF에 명시적으로 기술되어 있지 않으나, 기능 명세의 정합성을 위해 구현 시 반드시 고려해야 할 항목:

1. **미등록 시료 주문 시도**: 존재하지 않는 시료 ID로 주문 접수를 시도할 경우 오류 메시지를 표시하고 주문 생성을 거부한다.
2. **중복 시료 ID 등록**: 이미 존재하는 시료 ID로 재등록 시도 시 오류 처리한다.
3. **재고 정확히 일치하는 경우**: 현재 재고 = 주문 수량인 경우 "재고 충분"으로 간주하여 `CONFIRMED`로 전환한다 (부족분 0).
4. **수율 0 또는 음수 값 등록 방지**: 수율이 0이면 분모가 0이 되어 실 생산량 계산(ceil(부족분/수율))이 불가능하므로, 시료 등록 시 수율 값 검증(0 초과 1 이하)이 필요하다.
5. **음수/0 주문 수량**: 주문 수량은 반드시 1 이상의 정수여야 한다.
6. **REJECTED 주문에 대한 후속 처리 금지**: 거절된 주문은 승인, 생산, 출고 등 어떠한 후속 상태 전이도 발생하지 않는다.
7. **RESERVED가 아닌 주문에 대한 승인/거절 시도**: 승인/거절 메뉴에는 `RESERVED` 상태 주문만 노출되므로, 이미 처리된 주문에 대한 중복 처리를 원천 차단한다.
8. **CONFIRMED가 아닌 주문에 대한 출고 시도**: 출고 처리 메뉴에는 `CONFIRMED` 상태 주문만 노출되어야 한다.
9. **생산 큐 FIFO 순서 보장**: 여러 건이 동시에 PRODUCING으로 전환되어도 큐 등록 순서(승인 처리 순서)를 유지해야 한다.
10. **부분 재고 소진 후 생산**: 재고가 일부(0 초과) 있는 상태에서 부족한 경우, 기존 재고 전량을 우선 사용하고 부족분만 생산 요청한다 — 전체 주문 수량을 다시 생산하지 않는다.
